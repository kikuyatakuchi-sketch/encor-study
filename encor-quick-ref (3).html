Enterprise Architecture とは
▼

Cisco Enterprise Architectureは、ネットワークを機能別の「ブロック」に分割して設計する手法。
拡張性・セキュリティ・管理性を確保するためのモジュラー設計の考え方。

階層役割
Accessエンドデバイスの収容（スイッチポート・AP）
Distributionポリシー適用・ルーティング境界
Core高速転送・バックボーン（フィルタ不要）
💡 2-Tier vs 3-Tier
小規模はCollapsed Core（Distribution + Core統合）= 2-Tier
大規模は3-Tierで物理的にCore分離 → スケーラビリティ確保
📐
設計原則 6つ
▼
1
Hierarchy（階層化）
Access / Distribution / Core で役割分担
2
Modularity（モジュール化）
機能ブロック単位で独立設計・変更可能
3
Resiliency（冗長性）
FHRP、EtherChannel、冗長リンクで耐障害性
4
Flexibility（柔軟性）
将来の変更・拡張に対応しやすい設計
5
Security（セキュリティ）
各層でセキュリティポリシーを適用
6
Scalability（拡張性）
容量・ユーザー増加への対応能力

スイッチング方式の進化
▼

パケット転送の仕組みはProcess → Fast → CEFと進化。
現在のCiscoデバイスはCEFがデフォルトで最も高速。

方式仕組み速度
Process Switching全パケットをCPUで処理遅い
Fast Switching初回はCPU、以降はキャッシュ中間
CEFFIB + Adjacency Tableで即転送最速

TCAM (Ternary CAM)
▼

L3スイッチでCEFをハードウェア実装するための特殊メモリ。
CAMが「完全一致」のみなのに対し、TCAMは「0 / 1 / Don't Care」の3値で検索。

メモリ一致方式用途
CAM完全一致（Exact Match）MACアドレステーブル
TCAM3値一致（ワイルドカード可）ACL / QoS / ルーティング（FIB）
💡 TCAMの構造
Value: 照合するビットパターン
Mask: 照合対象のビット位置（Don't Careを指定）
Result: 一致時のアクション（permit/deny/転送先）
⚠️ TCAM枯渇に注意
TCAMはハードウェアリソースで容量に上限がある
ACLやルートエントリが多すぎると溢れ → ソフトウェア処理にフォールバック（性能劣化）

試験の落とし穴
▼
よく出るひっかけ
1. CEFはデフォルトで有効。無効化コマンドは no ip cef

2. FIBはRIB（ルーティングテーブル）から生成される（ARPではない）

3. Adjacency TableはARPテーブルから生成される（RIBではない）

4. TCAMは0/1/X（Don't Care）の3値。CAMは0/1の2値

5. TCAMが溢れるとソフトウェア処理になり性能が大幅低下

6. SDMテンプレート変更後はreloadが必要

7. Process SwitchingはCPU負荷が最も高いが、全機能が使える（デバッグ等）

8. CEFのPunt: ハードウェアで処理できないパケットをCPUに送ること

9. Glean: FIBでネクストホップが判明したがARP未解決 → ARP送信

10. CAMはMACアドレステーブルに使用。TCAMはACL/QoS/FIBに使用

QoS ってなに？
▼

ネットワーク上のトラフィックに優先順位をつけて品質を保証する仕組み。
帯域が足りないときに「何を先に通すか」を制御する。

モデル特徴
Best EffortQoSなし。全トラフィック平等（デフォルト）
IntServフロー単位で帯域予約。RSVP使用。スケーラビリティ低い
DiffServクラス単位で優先度設定。DSCPでマーキング。最も一般的
💡 ENCORではDiffServが主役
IntServはRSVPの概念だけ押さえればOK。
実務・試験ともにDiffServ（DSCP + PHB）が超頻出。
🏷️
マーキング（DSCP / CoS）
▼
フィールドレイヤビット数備考
CoS (802.1p)L23bit (0-7)802.1Qタグ内。トランクのみ
IP PrecedenceL33bit (0-7)ToSフィールド上位3bit（レガシー）
DSCPL36bit (0-63)ToSフィールド上位6bit（主流）

DSCP 主要値（PHB）

PHBDSCP名DSCP値用途例
EFExpedited Forwarding46音声（低遅延・低ジッタ）
AFAssured ForwardingAF11〜AF43重要データ（4クラス×3ドロップ）
CSClass SelectorCS0〜CS7IP Precedence互換
DFDefault0Best Effort
⚠️ AF命名規則 超頻出
AFxy → x = クラス(1-4)、y = ドロップ確率(1-3)
DSCP値 = 8x + 2y
例: AF31 = 8×3 + 2×1 = 26 / AF43 = 8×4 + 2×3 = 38

Policing vs Shaping
▼
項目PolicingShaping
超過時ドロップまたはリマークバッファリングして遅延送信
方向Inbound / OutboundOutboundのみ
遅延なし（即時判断）あり（バッファ待ち）

キューイング / 輻輳管理
▼

輻輳管理（キューイング）= 輻輳時の送信順序制御

方式特徴
FIFO先着順。デフォルト。QoSなし
PQ（Priority Queue）高優先度を常に先に送信。低優先度は飢餓の危険
WFQフロー単位で公平に帯域分配
CBWFQクラス単位で帯域保証（最も柔軟）
LLQCBWFQ + 厳密優先キュー。音声に最適

輻輳回避 = キューが溢れる前に予防的にドロップ

方式特徴
Tail Dropキュー満杯で末尾のパケットをドロップ。TCP同期問題
WREDランダムに早めにドロップ → TCP同期を防止
⚠️ LLQ = 音声のゴールドスタンダード
音声はLLQの優先キューに入れる。
WREDは音声トラフィックには適用しない（ドロップ不可のため）

MQC（Modular QoS CLI）
▼

QoSポリシーを3ステップで定義するCisco標準フレームワーク。

class-map
トラフィックの分類条件を定義（match文）
policy-map
各クラスへのアクション定義（set / police / shape等）
service-policy
インターフェースに適用（input / output）
! 1. 分類 class-map match-any VOICE match dscp ef match protocol rtp ! 2. ポリシー policy-map QOS-POLICY class VOICE priority percent 10 class class-default fair-queue ! 3. 適用 interface GigabitEthernet0/0 service-policy output QOS-POLICY
💡 match-all vs match-any
match-all: 全条件にAND一致（デフォルト）
match-any: いずれかの条件にOR一致
⚠️
試験の落とし穴
▼
よく出るひっかけ
1. DSCPは6bit（0-63）。IP Precedenceは3bit（0-7）。混同注意

2. EFのDSCP値は46。音声=EFはセット

3. AF計算: AFxy = 8x + 2y。AF41 = 34, AF21 = 18

4. ShapingはOutbound専用。Policingは双方向対応

5. WREDはTCPに効果的。UDPには効果なし（再送機能なし）

6. LLQの優先キューが使う帯域を制限しないと他クラスが飢餓状態

7. MQCの3要素: class-map → policy-map → service-policy

8. match-allがデフォルト。複数条件をOR結合したいならmatch-any

9. CoS はL2のみ。ルーティングされるとCoSは失われる（DSCPは維持）

10. Trust Boundaryは通常Access層（エッジ）に配置。不正マーキングを防止

RRM (Radio Resource Management)
▼

WLCが無線環境を自動最適化する機能。DCA / TPC / Coverage Hole Detectionの3本柱。

機能正式名役割
DCADynamic Channel Assignmentチャネルの自動割り当て（干渉回避）
TPCTransmit Power Control送信電力の自動調整
CHDCoverage Hole Detectionカバレッジ不足の検出・補正
💡 RRMの仕組み
各APがNDP（Neighbor Discovery Protocol）で周囲のRF環境を測定 → WLCに報告 → WLCが最適化計算 → APに指示
⚠️
試験の落とし穴
▼
よく出るひっかけ
1. CAPWAPのControlは常にDTLS暗号化。Dataはオプション

2. FlexConnectのConnected Mode: WLC経由 / Standalone Mode: WAN切断時ローカル処理

3. 2.4GHzの重複しないチャネルは1, 6, 11の3つのみ

4. 802.11acは5GHz専用。2.4GHzは使わない

5. 802.11axの新技術: OFDMA（帯域を細分化して同時通信）とBSS Coloring（干渉軽減）

6. AP DiscoveryでDHCPオプション43が最初に試行される

7. L3ローミングではMobility AnchorがクライアントのIPを維持

8. RRMのDCAはチャネル、TPCは電力。逆にする引っかけ多い

9. 802.11rはローミング高速化、802.11kはAP探索最適化（混同注意）

10. MIMOは802.11n〜、MU-MIMO（下り）は802.11ac Wave2から

VRFとは？
▼

Virtual Routing and Forwarding — 1台のルータ内に独立したルーティングテーブルを複数作成する技術。
物理的には同じ装置でも、論理的に完全分離されたネットワークを実現。

💡 用途

⚙️
VRF-Lite 設定
▼

MPLS不要でVRFの分離機能だけ使う構成。Enterprise環境で最も一般的。

定義フォーマット（新形式: VRF Definition）

vrf definition CUSTOMER-A rd 65000:1 ! address-family ipv4 route-target export 65000:1 route-target import 65000:1 exit-address-family ! interface GigabitEthernet0/1 vrf forwarding CUSTOMER-A ip address 10.1.1.1 255.255.255.0
⚠️ 超重要：設定順序
vrf forwardingをIFに適用するとIPアドレスが消える！
必ず VRF割当 → IPアドレス設定 の順番で。
項目説明
RD
(Route Distinguisher)		VRFのルートを一意に識別するタグ
形式: ASN:nn or IP:nn（例: 65000:1）
RT
(Route Target)		MPLS VPNでルートのimport/exportを制御
VRF-Liteでは設定不要な場合も
💡 旧形式 vs 新形式
旧: ip vrf（IPv4のみ対応）
新: vrf definition（IPv4 + IPv6 デュアルスタック対応）
ENCORでは新形式を推奨

▼

メトリック計算
▼

Classic メトリック（K値）

Metric = 256 × (K1×BW + K2×BW/(256-Load) + K3×Delay) × K5/(K4+Reliability) デフォルト: K1=1, K2=0, K3=1, K4=0, K5=0 → 実質: 256 × (BW + Delay)
BW10⁷ / 経路上の最小帯域幅(Kbps)
Delay経路上の遅延合計 / 10 (μs)

Wide Metric（Named Mode）

64bit対応で高速リンクの差別化が可能。
rib-scale 128 で64bit → 32bit変換してRIBに挿入。

💡 K値の罠
K値不一致 → ネイバーにならない
K値変更はmetric weightsで設定（通常は変更不要）

▼

▼
タイプ説明用途

ip nat pool MYPOOL 203.0.113.1 203.0.113.10 netmask 255.255.255.0 access-list 1 permit 192.168.1.0 0.0.0.255 ip nat inside source list 1 pool MYPOOL

▼

PIM（Protocol Independent Multicast）
▼

ルータ間でマルチキャスト配信ツリーを構築するプロトコル。
「Protocol Independent」= どのユニキャストルーティングプロトコルとも連携可能。

モードツリー用途
PIM Dense ModeSource Tree（最短パス）受信者が多い環境（Push型）
PIM Sparse ModeShared Tree（RP経由）エンタープライズ標準（Pull型）

Dense Modeの動作

Flood
全IFにマルチキャストをFlood
Prune
不要なルータがPrune送信 → 枝刈り

3分ごとにFlood/Prune繰り返し → 非効率（実務ではほぼ使わない）

Sparse Modeの動作

Join
受信者側ルータがRP（ランデブーポイント）にJoin送信
Shared Tree (*,G)
RP経由の共有ツリーでデータ配信開始
SPT Switchover
閾値超過で最短パスツリー(S,G)に切替（オプション）
🎯

5. RPFチェックはユニキャストルーティングテーブルを参照する

6. (*,G) = Shared Tree（RP経由）/ (S,G) = Source Tree（最短パス）

7. Auto-RPはCisco独自、BSRはRFC標準

8. ip multicast-routing をグローバルで有効にしないと何も動かない

9. Dense ModeはFlood & Pruneで非効率 → 実務ではSparse Mode

10. 224.0.0.0/24のLink LocalアドレスはTTL=1でルーティングされない

CoPP とは
▼

Control Plane Policingはルータ/スイッチのコントロールプレーン（CPU）宛てのトラフィックをレート制限して保護する機能。

プレーン役割脅威
Data Planeユーザトラフィック転送ACLで保護
Control Planeルーティング・管理プロトコル処理DoS攻撃でCPU枯渇
Management PlaneSSH/SNMP等の管理アクセス不正アクセス
⚠️ なぜCoPPが必要？
ルーティングプロトコル（OSPF/BGP等）やSSH/SNMP等はCPU処理が必要
大量のトラフィックがCPUに到達するとルーティング障害・管理不能に
CoPPで重要トラフィック優先・不正トラフィック制限

CPPr（Control Plane Protection）
▼

CoPPの拡張版。コントロールプレーンを3つのサブインターフェースに細分化して、より細かく制御。

サブIF対象例
Hostルータ自身宛てSSH, Telnet, SNMP

試験の落とし穴
▼
よく出るひっかけ
1. CoPPはMQC（class-map / policy-map / service-policy）を使用。QoSと同じ仕組み

2. CoPPの適用先はcontrol-plane、方向はinputのみ

3. CoPPが保護するのはCPUへのトラフィック。Data Planeは対象外

4. ルーティングプロトコル（OSPF/BGP等）は高優先度で許可すべき

6. CPPrはHost / Transit / CEF-Exceptionの3分割。CoPPより精密

8. CoPPなしだとICMP flood等でCPU 100% → ルーティング崩壊のリスク

9. Management Planeの保護はMPP（Management Plane Protection）が別途ある

10. CoPPのmatch-allは全条件一致、match-anyはいずれか一致

TrustSec 概要
▼

Cisco TrustSecはIPアドレスに依存しないアクセス制御を実現するフレームワーク。ユーザ/デバイスにSGT（Security Group Tag）を割り当て、SGT間の通信をSGACLで制御する。

概念説明
SGTSecurity Group Tag。16bit（0-65535）のタグ。ユーザの役割を識別
SGACLSGT間のトラフィック制御ポリシー（送信元SGT × 宛先SGT）
SXPSGT eXchange Protocol。SGT⇔IPマッピングをTCP経由で伝搬
ISESGT割当＋SGACL定義の中央管理サーバ
💡 なぜTrustSec？
従来のACLはIPアドレスベース → ユーザが移動するとACL書き換え必要。
TrustSecはロール（役割）ベース → ユーザがどこにいてもSGTは同じ → ポリシーの一貫性
🔄

▼
方式説明用途

SGACL（ポリシー制御）
▼

SGACLは送信元SGT × 宛先SGTのマトリクスでアクセス制御を定義。ISEで一元管理し、ネットワーク機器にプッシュ。

送信元＼宛先SGT 30 (Server)SGT 40 (Internet)
SGT 10 (社員)permit allpermit all
SGT 20 (ゲスト)deny allpermit http https
SGT 5 (IoT)permit tcp 443deny all
! SGACL定義（ISE or ローカル） cts role-based permissions from 20 to 30 GUEST-TO-SERVER ! ! SGACLの内容 ip access-list role-based GUEST-TO-SERVER deny ip ! ! SGACLのEnforcement有効化 cts role-based enforcement ! ! IF単位でも有効化 interface GigabitEthernet1/0/1 cts role-based enforcement
⚠️ SGACL 適用ポイント
SGACLは宛先（Egress）側のスイッチで評価・適用される。送信元側ではない！
ISEからSGACLポリシーがダウンロードされるため、ISEとの連携が必須。
⚠️

▼
よく出るひっかけ
1. TrustSecはIPアドレスではなくSGT（ロール）ベースのアクセス制御

2. SGTは16bitのタグ。ISEが認証時に動的割当（手動も可）

3. SGACLは宛先側で評価される。送信元側ではない

4. SXPはTCPベース。Speaker(送信) / Listener(受信)のペア

5. インラインタギングはL2フレームにCMD(Cisco Meta Data)として挿入

6. SXPはインラインタギング非対応機器の橋渡し用

7. TrustSecの中央管理はISE。SGT割当 + SGACL定義 + ポリシー配布

8. cts role-based enforcementコマンドでSGACL適用を有効化

9. SGT=0はUnknown（未分類）。デフォルトポリシーが適用される

10. TrustSecはACLと異なりトポロジ非依存 → ユーザ移動してもポリシー維持

802.1X 概要
▼

IEEE 802.1Xはポートベースのネットワークアクセス制御(PNAC)。未認証のデバイスはネットワークに接続不可。認証が成功して初めてポートが開放される。

コンポーネント役割具体例
Supplicant認証を要求する側（クライアント）PC, スマホ
Authenticator認証の仲介（アクセス制御）スイッチ, WLC
Authentication Server認証の判断ISE, RADIUSサーバ
EAPOL-Start
Supplicant → Authenticator: 認証開始要求
EAP-Request/Identity
Authenticator → Supplicant: ID要求
EAP over RADIUS
Authenticator → Auth Server: EAPメッセージをRADIUSでカプセル化転送
RADIUS Access-Accept
認証成功 → SGT / VLAN / dACL等の認可属性を返却
ポート開放
Authenticatorがポートを Authorized 状態に変更。通信許可
💡 EAPOL vs RADIUS
Supplicant ⇔ Authenticator: EAPOL（EAP over LAN）= L2プロトコル
Authenticator ⇔ Auth Server: RADIUS = L3プロトコル（UDP 1812/1813）
⚙️
802.1X ポート状態 & Host Mode
▼

ポートの認証状態

状態説明
Unauthorized認証前。EAPOLのみ通過（他は全てブロック）
Authorized認証成功。全トラフィック通過
Force-Authorized常に開放（デフォルト）。802.1X無効時
Force-Unauthorized常にブロック

Host Mode

モード動作用途
Single-Host1ポート1MAC。2台目以降はブロック1対1環境

⚠️ IP Phone + PC構成
Multi-Domain Authentication (MDA)を使用。
Data Domain: PCが802.1X認証
Voice Domain: IP Phoneが認証（CDPまたはLLDP検出）
それぞれ別VLAN（Data VLAN + Voice VLAN）に割り当て

Change of Authorization (CoA)
▼

CoAはRADIUS拡張機能（RFC 5176）。認証後にISEからポリシーを動的変更できる。

CoA タイプ動作用途

CoA-Reauthenticate再認証を強制ポリシー変更の即時反映
CoA-Terminateセッション強制終了不正端末の即時遮断
Bounce-Portポートを一時Down/UpVLAN変更を反映（DHCP再取得）
! CoA受信設定 aaa server radius dynamic-author client 10.1.1.100 server-key SharedSecret port 1700
⚠️ CoA の方向に注意
通常のRADIUSはNAS → サーバ方向。
CoAは逆でサーバ（ISE）→ NAS（スイッチ）方向。ポート1700（or 3799）。
dynamic-authorコマンドでCoAクライアント（ISE）を登録。

YANG データモデル
▼

概念説明
moduleYANGの最上位単位。名前空間を定義
container子ノードをグループ化する箱（1つだけ存在）

leaf値を持つ末端ノード（string, uint32等）
leaf-list同じ型の値を複数持てるリーフ

💡 YANG モデルの種類

OpenConfig: ベンダー中立の業界標準モデル

試験の落とし穴
▼
よく出るひっかけ
1. YANGはデータモデルであり、プロトコルではない。NETCONF/RESTCONFがプロトコル

2. JSONのキーはダブルクォート必須。{'key':'val'}はPython dictでありJSON不正

4. YANG containerは1つのみ、listは複数インスタンス可能

5. listには必ずkeyが必要。keyで各インスタンスを一意に識別

6. XMLの名前空間（xmlns）はYANGモジュールに対応

7. RESTCONFはJSONとXML両方対応。NETCONFはXMLのみ

8. YAML の --- はドキュメント開始、... はドキュメント終了

REST API の基本
▼

REST（Representational State Transfer）はHTTPベースのAPI設計スタイル。ネットワーク自動化ではRESTCONFやDNA Center APIで使用。

HTTPメソッドCRUD動作冪等性
GETReadデータ取得✅ 冪等
POSTCreate新規作成❌ 非冪等
PUTReplace全体置換（なければ作成）✅ 冪等
PATCHUpdate部分更新✅ 冪等
DELETEDelete削除✅ 冪等
💡 冪等性（Idempotent）とは
同じリクエストを何回送っても結果が同じこと。
POSTだけ非冪等 → 同じPOSTを2回送るとリソースが2つ作られる
PUT/PATCH/DELETEは何回送っても結果は同じ

コード意味場面

Ansible の基本
▼

AnsibleはPython製の構成管理・自動化ツール。エージェントレスでSSH/NETCONF経由でデバイスを操作する。

特徴説明
エージェントレス対象ノードにソフト不要。SSH/NETCONFで接続
冪等性何回実行しても同じ結果になる（あるべき姿に収束）
YAML記述Playbook・InventoryはYAMLで記述
Push型コントロールノードから対象へプッシュ実行
Python製コントロールノードにPython必須（Linux/Mac）
💡 Ansible vs Puppet / Chef
Ansible: エージェントレス・Push型・YAML
Puppet: エージェント必要・Pull型・独自DSL（Puppet言語）
Chef: エージェント必要・Pull型・Ruby
→ ENCORではAnsibleが中心。「エージェントレス」が最大の特徴

Playbook 実例
▼

Inventory ファイル（hosts.yml）

# inventory: hosts.yml all: children: routers: hosts: R1: ansible_host: 192.168.1.1 R2: ansible_host: 192.168.1.2 vars: ansible_network_os: ios ansible_connection: network_cli ansible_user: admin ansible_password: cisco123

Playbook（configure_ntp.yml）

--- - name: Configure NTP on routers hosts: routers gather_facts: no tasks: - name: Set NTP server cisco.ios.ios_config: lines: - ntp server 10.1.1.100 - ntp server 10.1.1.200 - name: Save configuration cisco.ios.ios_config: save_when: modified - name: Show NTP status cisco.ios.ios_command: commands: - show ntp status register: ntp_output - name: Display NTP output debug: var: ntp_output.stdout_lines
⚠️ Playbook構文の注意
--- でYAMLドキュメント開始
gather_facts: no → ネットワーク機器では必須（デフォルトだとエラー）
ansible_connection: network_cli → NW機器はSSH CLI接続

💡 Push vs Pull
Push型（Ansible）: コントロールノードが能動的に設定を送信
Pull型（Puppet/Chef）: エージェントが定期的にマスターから設定を取得
→ AnsibleはPush型なので即時実行に向く。Puppetは継続的適用に向く

▼
よく出るひっかけ
1. Ansibleはエージェントレス。Puppet/Chefはエージェント必要

2. AnsibleはPush型、Puppet/ChefはPull型

3. PlaybookはYAMLで記述。Puppetは独自DSL、ChefはRuby

7. registerでタスク結果を変数に保存 → debugで表示

8. Jinja2: {{ }}は変数展開、{% %}はロジック。AnsibleのTemplate機能で使用

9. 冪等性: 同じPlaybookを何回実行しても結果が同じ。changed=0が理想

10. Ansibleの実行コマンド: ansible-playbook -i hosts.yml playbook.yml
